Some experiences on parallel code tuning with MPI+OpenMP¶

The main goal is to show some configuration options available for OpenMPI (>=3.0) (compiled with GCC 6.4.0) an OpenMP to control the CPUs (virtual or physical cores) where the code will run.

These experiences have been performed in two identycal machines, each one with two processors Intel Xeon CPU E5-2680v4 @ 2.40GHz (a total of 28 physical cores for each machine) and 64 Gb of RAM. The sole difference between these machines is that in one of them the Hyper-Threading was deactivated in the BIOS. Thus, the operational system (CentOS 7) sees 28 processors in one machine and 56 processors in the other.

First code (T1)¶

The first code (a simple matrices multiplication) is written as:

In [ ]:
/** CODE T1 **/
*t = omp_get_wtime();
#pragma omp parallel for private(j,k)
for (i=0; i<N; i++)
{
for (j=0; j<N; j++)
  for (k=0; k<N; k++)
    c1[i*N+j] += a[i*N+k]*b[j*N+k];
}
*t = omp_get_wtime(); t++;

Notice that this code can be related to the operation A*transpose(B). This is done in order to assume the second multiple (b) to be aligned in memmory. In fact, these variables are declared as:

In [ ]:
//memmory alignment: in runtime, a,b,c1 will be aligned in chunks of 64 bits (minimum)
double *a  = (double*)_mm_malloc(sizeof(double)*N*N,64);
double *b  = (double*)_mm_malloc(sizeof(double)*N*N,64);
double *c1 = (double*)_mm_malloc(sizeof(double)*N*N,64);

and initialized to:

In [ ]:
/** CODE T0 **/
for (i=0; i<N*N; i++)
{
    a[i] = sin((double)(i%20));
    b[i] = cos((double)(i%20));
    c1[i] = 0.;
    c2[i] = 0.;
}

After compiling the code by

$ mpicc example.c -o example -lm -fopenmp -O3 -mtune=native -DCLS=$(getconf LEVEL1_DCACHE_LINESIZE)

One runs the code passing the matrices order (N) by the first argument. For instance, for N=1000 one calls:

$ ./example 1000
T0 0.075 T1 1.036 T2 0.259

where T0, T1, and T2 correspond to the computational time (in seconds) for the initialization, CODE T1 and CODE T2 (this last one to be presented next).

Second code (T2)¶

Taken from here (see section 6.2.1 and Appendix A.1), the second code (AVX-vectorized matrices multiplication) is written as:

/** CODE T2 **/
#define SM (CLS / sizeof (double))
double *rres;
double *rmul1;
double *rmul2;

*t = omp_get_wtime();
#pragma omp parallel for private(j,k,i2,j2,k2)
for (i = 0; i < N; i += SM)
    for (j = 0; j < N; j += SM)
      for (k = 0; k < N; k += SM)
        for (i2 = 0, rres = &c2[i*N+j], rmul1 = &a[i*N+k]; i2 < SM;
             ++i2, rres += N, rmul1 += N)
        {
          _mm_prefetch (&a[8], _MM_HINT_NTA);
          for (k2 = 0, rmul2 = &b[k*N+j]; k2 < SM; ++k2, rmul2 += N)
          {
            __m128d m1d = _mm_load_sd (&rmul1[k2]);
            m1d = _mm_unpacklo_pd (m1d, m1d);
            for (j2 = 0; j2 < SM; j2 += 2)
            {
              __m128d m2 = _mm_load_pd (&rmul2[j2]);
              __m128d r2 = _mm_load_pd (&rres[j2]);
              _mm_store_pd (&rres[j2],
              _mm_add_pd (_mm_mul_pd (m2, m1d), r2));
            }
          }
        }

A simple run¶

Here show the basic tools used for the analysis performed in this document. First, one is able to create a mapping processor_number to core_id. When Hyper-Threading is active, two different processors number are mapped into a single core id. In the current context, the core id is labeled by two parameters: core_id=(physical id, core id), which can be obtained from the file /proc/cpuinfo. Doing so it is obtained a mapping like:

physical id = 0 physical id = 1 physical id = 0 physical id = 1
processor core id processor core id processor core id processor core id
0,28 0 14,42 0 7,35 8 21,49 8
1,29 1 15,43 1 8,36 9 22,50 9
2,30 2 16,44 2 9,37 10 23,51 10
3,31 3 17,45 3 10,38 11 24,52 11
4,32 4 18,46 4 11,39 12 25,53 12
5,33 5 19,47 5 12,40 13 26,54 13
6,34 6 20,48 6 13,41 14 27,55 14

For example, if our program runs 4 threads, setting them to run on the processors 0 to 3 will allow them to run on 4 different physical cores. While setting them to run on the processors 0, 28, 1 and 29, will force them to run on two physical cores (two threads each).

The GOMP_CPU_AFFINITY variable¶

Running a code parallelized with MPI/OpenMP without configuring the threads distribution will make the OS and the process manager (Slurm, PBS, etc). One option available for the user to set this configuration the the environment varible GOMP_CPU_AFFINITY link). It allows to bind the different threads of a program compiled with the OpenMP library.

Example: 28 threads running on physical cores¶

Setting

$ export OMP_NUM_THREADS=28

and doing

$ export GOMP_CPU_AFFINITY="0-27"
$ ./example 3000

returns

T0 0.036 T1 1.652 T2 0.388

while

$ export GOMP_CPU_AFFINITY="0-13 28-41"
$ ./example 3000

returns

T0 0.059 T1 1.542 T2 0.618
